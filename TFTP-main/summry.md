# TFTP服务端总结

## 大概流程
设置好服务器套接字，绑定好服务器地址信息结构体。循环监听客户端的请求报文

当监听到客户端发送过来的请求报文后，创建用于通信的套接字。将其设置好接收时间限制，避免无限拥塞等待。然后使用`connect`连接到客户端地址。这样一来既可以简化后续操作，使用`send recv`而不是`sendto recvfrom`，还可以过滤不是对端的信息。注意，这个和tcp连接的connect并不相同。

下一步可以把用于通信的套接字，客户端的请求报文等信息一起传参给线程体函数。这样以来，主线程中监听客户端发来的请求信息。多线程中完成数据的上传和下载等操作。因为在线程体中还可能涉及到类的操作，所以可以把当前类的指针也传递过去。

```c++
struct tftp_work{
    int pfd = -1; //会话套接字
    std::string buff; //客户端的请求报文
    sockaddr_in cin; //客户端地址信息结构体
    socklen_t socklen; //结构体长度
    tftp_server* server;
};
```

其中 `server=this`。

在线程体函数中解析传进来的参数，解析客户端的请求报文，把操作码和文件名称解析出来。之后就可以根据操作码来调用对应的函数，完成响应的操作。

在处理下载请求时，根据请求的文件名称打开对应的文件，读取文件内容组装报文发送给客户端。如果读取到的内容长度小于512字节，说明此报文为最后一个。可以准备结束操作了。

在处理上传请求时，要先发送一个ACK0给客户端表示准备好接受上传的数据。然后接受客户端发送的数据并写入对应的文件，组装ack报文回复客户端。如果客户端发送的数据端长度小于512字节。说明已经是最后一块数据块。

## 遇到的问题

1. 在设计类时，服务器IP地址这种属于整个服务器的类型可以设置为`static`静态变量。但是，因为每一个类都会初始化一份变量，为了避免重复定义的错误。不能在类中直接声明并定义。要类内声明，类外定义。或者使用`inline`关键字。

2. 关于友元。在线程体函数中处理客户端请求时，可能会用到类中被封装的私有函数，此时可以在类中将线程体函数声明为友元函数。注意，在类中声明友元函数仅仅是访问权限的声明，还需要在类外再进行一次函数的声明。

3. 可以把this指针传递给线程体函数，实现在函数中对这个类的函数调用。

4. 可以使用string来组装报文或作为接受缓冲区。如果作为缓冲区，要记住使用`resize`再次更新实际读取的长度。成员函数`c.str()`返回的是c风格的指针。`data()`返回的则是可以写的指针类型。

5. 解析/组装2字节的字段

```c++
//使用static_cast手动按字节拆分和组装
#include <string>
//解析操作码。先确保为8位，把两个数字拼接到一起
uint16_t opcode = static_cast<uint16_t>(static_cast<uint8_t>(buff[0])<<8)|static_cast<uint16_t>(static_cast<uint8_t>(buff[1])&0xff);
//组装块编号
std::string packet;
packet+=static_cast<char>(blockNum>>8);
packet+=static_cast<char>(blockNum&0xff);
```
或者使用`reinterpret_cast`从地址的角度上重新解释。

```c++
    //把块编号转变为网络字节序
    uint16_t n = htons(static_cast<uint16_t>(nextBlockNum));
    //因为块编号的类型是uint16_t，需要使用reinterpret_cast
    //直接按照字节的角度添加两个字节。
    //重新解释地址，以char*的角度来看
    packet.append(reinterpret_cast<const char*>(&n),2);
```
